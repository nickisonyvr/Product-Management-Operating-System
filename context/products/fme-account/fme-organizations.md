# FME Organizations

> Feature context for **FME Organizations** in the **FME Account** product. Linked from project `input-references.md` files.

## Feature Overview

FME Organizations is the enterprise administration layer built within FME Account that lets Organizational Administrators self-serve their FME deployments — generating, assigning, and reassigning serial numbers for FME Form and FME Flow without involving Safe staff. It is built on an identity and organization foundation: organizations are linked to HubSpot subscriptions, members are managed by Org Admins, and domain auto-association keeps rosters current as teams grow.

Today, provisioning a serial is a manual, email-driven process routed through an Account Manager — with no customer-facing workflow, no link between a serial and the person using it, and no governance trail when staff leave. FME Organizations replaces that with a self-serve portal. The London MVP (targeting March 2027) focuses on fixed (node-locked) subscription customers; floating-license support is Phase 2. A sibling PRD covers Usage Observability.

**Core Value**: For the first time, subscription customers can self-administer their FME deployments — managing licensing, users, observability, and product settings without involving Safe.

---

## User Jobs

1. **Generate a serial for a new hire** — Org Admin needs to onboard a new team member; wants to generate and assign an FME Form serial in minutes without emailing an Account Manager.
2. **Reassign a serial when someone leaves** — Org Admin needs to reclaim a serial from a departing employee and assign it to their replacement, with a full audit trail.
3. **Recover my own serial** — An end user needs to find their assigned serial without digging through old email or asking their admin.
4. **See the full licensing roster** — Org Admin wants a single view of all serials, who holds them, and which are unassigned, so they can govern their subscription actively.
5. **License an FME Flow instance** — Org Admin needs to generate a serial for a Flow server deployment without involving Safe.

## Key Workflows

### Workflow 1: Generate and assign a serial (FME Form)

1. Org Admin opens the Serials page in FME Account.
2. Clicks "New Serial", selects FME Form, and selects an assignee from existing members — or invites a new member inline.
3. Generation is submitted asynchronously; modal shows "Generating — this may take a few minutes."
4. Serial is generated by SET and stored; assignee receives an automated email with the serial.
5. Serials page updates to show the new serial with "Active" status and the assignee's name.

### Workflow 2: Reassign a serial

1. Org Admin locates the serial on the Serials page (search by name, email, or serial number).
2. Selects "Reassign" and chooses a new assignee — an existing member, an invited member, or unassigned.
3. Reassignment is recorded as an event in the serial's audit history; previous assignee remains visible in the log.

## Key Concepts / Data Model

| Concept | Description |
|---------|-------------|
| **Organization** | The top-level entity in FME Account, linked 1:1 to an active FME Subscription via a HubSpot Company ID. Created by Safe during Order Processing — not self-serve in v1. |
| **Membership** | The relationship between an FME Account user and an Organization. Roles: Org Admin (full control) and Member (read-only, own serial only). Removed memberships persist as stubs for audit continuity. |
| **Serial** | A tracked licensing entity owned by an Organization. Form serials require an assignee; Flow serials do not. Each serial has a status: Pending generation, Active, or Unassigned. |
| **Subscription** | The active commercial entitlement linked to an Organization, sourced from HubSpot via SET. Gates serial generation — generation is blocked if the subscription is inactive. |
| **Domain auto-association** | When a new FME Account user signs up with an email matching an Org's configured domain, they are offered auto-add to that Org (or auto-approved, if configured). Multiple domains per Org are supported. |
| **Audit log** | An immutable record of all write actions (generate, reassign, invite, remove member) with actor, timestamp, target, and Org context. |

## Edge Cases / Constraints

- **Invited-but-not-accepted assignee** — A serial can be associated with an invited member before they accept. The serial email is sent only after the invite is accepted; the Serials page shows "Pending invite acceptance" until then.
- **Pre-existing serials** — Serials issued before FME Organizations existed appear in the Serials page as unassigned, surfaced via SET. Org Admins can assign them via the standard reassignment workflow.
- **SET API unavailability** — If SET is unreachable during a subscription validity check, FME Account falls back to last-known-good cached state and surfaces a soft warning in the admin UI. Generation is blocked only on confirmed inactivity, not on a failed check.
- **Subscription sync cadence** — Subscription validity is refreshed daily (batch) plus on-access when an Org Admin opens the licensing area. Not real-time — edge cases exist where a just-expired subscription briefly appears active.
- **Serial obfuscation** — Serials are sensitive (exchangeable for license files). Only the last 4 characters are shown by default; click-to-expose reveals the full serial.
- **Bulk serial generation** — Explicitly out of scope. Mass-onboarding is handled by internal scripted tooling, not the customer UI.
- **Floating licenses** — Not supported in v1. Fixed (node-locked) subscription customers only. Phase 2 pending resolution of strategic and technical open questions.

## Integrations / Dependencies

| System / Feature | Direction | Purpose |
|------------------|-----------|---------|
| **SET (licensing system)** | Both | Source of truth for serials and subscription state. FME Account calls SET to generate serials, list serials by Org, check serial status, and validate subscription validity. SET is also building license-file generation event history used here. |
| **HubSpot** | In | Source of commercial subscription data (via SET). HubSpot Company ID is the trusted external reference linking an Org to its Subscription. FME Account does not write back to HubSpot in this PRD. |
| **Auth0** | In | Underpins FME Account authentication. Domain auto-association shares plumbing with SSO domain configuration but is a separate feature. |
| **FME Form** | Out | Consumes Org-generated serials for activation. In-product login enforcement (requiring FME Account login at activation) is a future phase dependency. |
| **FME Organizations: Usage Observability** | Both | Sibling PRD. The serial-first identity model and membership data established here provide the clean attribution data the Observability PRD depends on for usage dashboards and HubSpot usage push. |

## Related Features

- **FME Organizations: Usage Observability** — Sibling PRD covering customer-facing usage dashboards and HubSpot usage push. Depends on the serial-to-user identity model established here.
- **FME Account SSO** — Domain auto-association shares configuration plumbing with SSO; SSO is gated more carefully and is a separate feature.

## Out of Scope

- **Perpetual license management** — Legacy perpetual contracts remain on existing internal workflows.
- **Floating-license pool management** — Phase 2; pending strategic and technical open questions.
- **In-product login enforcement** — Requiring FME Account login at FME Form activation for Org-generated serials. Future phase (2027).
- **Strict named-user enforcement** — Login requirement will enforce Org membership, not exact assignee match. Strict match is a future-version candidate.
- **FME Flow Hosted licensing** — Flow Hosted does not use serials and is not actionable in v1.
- **Reseller / Partner views** — Managing licensing on behalf of clients is out of scope.
- **Self-serve Org creation** — Organizations are provisioned by Safe via Order Processing.
- **Bulk serial generation** — Mass-onboarding handled by internal scripted tooling.
- **In-product license file generation** — Customers continue to generate `.fmelic` files through existing in-product flows.
- **Audit log surfaced to Org Admins** — Audit log is internal-only in v1; customer-facing is a v2 candidate.
